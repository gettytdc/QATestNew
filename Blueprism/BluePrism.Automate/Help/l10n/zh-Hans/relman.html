Blue Prism 发行版管理器

简介

发行版管理器

概述

Blue Prism 包含一种机制，通过该机制，Blue Prism 配置中的各组件可在不同 Blue Prism 环境之间传输

发行版管理器允许将捆绑组件定义为“发行包”。然后，可以将创建的发行包作为一个“发行版”导出到文件中。

发行包

发行包提供了一种用于收集发行版所含元素的列表的机制。它在 Blue Prism 客户端的发行版管理器中创建。

发行包由名称、描述及其内容组成。发行包的内容可以由以下任一或所有组件组成：—

  • 流程
  • 对象
  • Web 服务定义
  • 流程组
  • 环境变量
  • 凭据
  • 工作队列
  • 计划
  • 字体

创建发行包后，您可以从中创建一个发行版。

可以在发行版管理器中修改或删除发行包。

如果一个发行包中包含已删除的组件，则对该组件的引用会从发行版中删除

发行版

发行版代表特定时间点的发行包—更实际的说，发行版是包含保存在其中的发行包的所有内容的文件。

它由名称、一些发行版说明和发行包的详细内容组成—即，如果发行包中包含组件列表，则发行版会包含这些组件的数据。

创建

创建发行版时，文件将保存到硬盘驱动器，并会在数据库中保存一个条目，其中包含有关发行版的详细信息，例如其名称、发行版说明、用户、创建日期/时间及创建发行版时的内容。请注意，如果发行包内容发生更改或删除了发行包中的组件，则系统上的发行版内容保持不变。

发行版被保存到文件，通常带有 .bprelease 扩展名。然后，可以将其导入其他 Blue Prism 4.1+ 环境中。在发行版被导入某个环境中时，也会在该发行版的数据库中记录一个条目。这个记录与上述发行版类似（除了创建用户和创建日期/时间），详细信息中将包括导入用户和导入日期/时间。

导入

发行版可以从发行版管理器中导入，也可以从 Blue Prism 客户端的“文件”菜单中选择“导入”来导入。

不同的组件使用不同的规则来确定它们应该如何对目标系统中的冲突做出反应—大多数组件需要某种形式的用户输入来指示在发生这种冲突时应该采取的操作。

导入规则

下面给出了定义如何处理每个受支持组件的规则：—

流程
  • 如果目标环境不包含与传入流程具有相同 ID 或名称的流程/VBO，则无需任何进一步的用户干预即可导入。
  • 如果目标环境包含与传入流程具有相同 ID 的流程/VBO，则会提示用户：—
    • 覆盖现有流程（默认）—或—
    • 使用新 ID 保存流程。
    如果选择后者且现有流程/VBO 与传入流程具有相同的名称，则系统会提示用户输入流程的新名称。
  • 如果目标环境包含与传入流程具有相同名称的流程/VBO，则系统会提示用户：—
    • 重命名传入流程（默认）—或—
    • 覆盖现有流程/VBO—或—
    • 重命名现有流程。
    如果选择重命名，则系统会提示用户提供新名称来重命名流程/VBO。
可视业务对象
  • 处理 VBO 的方式与流程相同。
Web 服务定义
  • 如果目标环境不包含与传入服务具有相同名称的 Web 服务，则无需任何进一步用户干预即可导入。
  • 如果目标环境包含与传入服务具有相同名称的 Web 服务，则系统会提示用户：—
    • 覆盖现有 Web 服务（默认）
    • 为导入 Web 服务选择新名称
    • 为现有 Web 服务选择新名称
    如果选择重命名，则系统会提示用户为相关 Web 服务提供一个新名称。
流程组
  • 传入流程组与目标环境中具有相同名称的现有流程组合并。如果传入组包含不属于现有组的导入流程，则会在传入流程被导入后添加它们。
环境变量
  • 如果目标系统中不存在具有相同名称的环境变量，则创建发行版时会在目标环境中设置源环境中的变量及其值。
  • 如果在目标环境中找到具有相同名称的环境变量，则会使用传入描述和数据类型对其进行更新。如果现有值无法转换为传入数据类型，则会在目标环境中设置传入值，否则将保留当前值。
凭据
  • 如果目标环境中不存在具有相同名称的凭据，则系统会提示用户提供要在凭据中设置的用户名和密码，并创建该凭据。

  • 如果目标环境中存在具有相同名称的凭据，则传入凭据会与其合并，以便在现有凭据中设置传入描述，且与该凭据关联的任何传入流程都会在目标系统上关联。

    导入同名凭据不会影响现有的用户名和密码。

  • 无论凭据存在与否，用户都可以选择不在导入流程中导入凭据。

注意： 在导入任何凭据之前，必须在目标环境中使用默认加密方案，否则将发生错误且导入操作失败。

工作队列
  • 如果目标环境中不存在具有相同名称的工作队列，则会使用传入详细信息创建一个工作队列。
  • 如果目标环境中存在具有相同名称的工作队列，则其详细信息（键字段和最大尝试次数）将被传入队列覆盖。队列中的任何项目都不受这些更改的影响。
计划
  • 如果不存在与导入的计划具有相同名称的计划，则会在目标环境中创建不包含计划会话的计划。
  • 如果目标环境中存在具有相同名称的计划，则使用传入计划定义覆盖其描述、时间数据和任务结构。请注意：—
    • 请注意，如果存在与随同计划导入的任务具有相同名称的任务，则会覆盖其描述及其“成功时”和“异常时”设置，但会保留计划会话列表。
    • 另请注意，如果现有计划“已停用”，则它不会因为导入计划而被启用，尽管其数据会发生变化—它必须在需要时手动“取消停用”。
日历
虽然日历不能显式添加到发行包中，但如果某个计划需要日历才能运行，则将随同任何生成的发行版中的计划提供日历。
  • 如果目标环境中不存在具有相同名称的日历，则会在目标环境中创建
  • 如果目标环境中存在具有相同名称的日历，则会使用传入日历数据将其覆盖。
字体
  • 如果目标系统上不存在具有相同名称的字体，则会使用传入详细信息创建。
  • 如果存在具有相同名称的字体，则会使用传入字体将其覆盖。

验证

使用系统中提供的机制，可以根据发行版的来源环境对其进行验证—这将检查创建发行版以来环境是否发生任何更改。

传统导入/导出

在提供发行版管理器之前，Blue Prism 支持单个流程或可视业务对象的导入和导出。从这些早期 Blue Prism 版本创建的文件仍然可以使用与导入发行版相同的机制导入—只需在“文件打开”对话框中更改文件筛选器，以查找 *.xml 文件而不是 *.bprelease 文件。

如果您需要导出流程/可视业务对象，以便在旧版本的 Blue Prism 中导入它，您仍然可以像以前一样，通过在 Process Studio/Object Studio 中打开流程/对象，然后选择“文件 | 导出 | 此流程”或“文件 | 导出 | 此页面”导出流程或页面来实现此目的。
