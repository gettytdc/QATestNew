オブジェクトスタジオ

はじめに

このページでは、Blue Prismのオブジェクトスタジオ機能を使用する際のよくある問題と推奨される解決方法について説明します。

アプリケーション要素をスパイした直後に、Blue Prismがその要素をハイライトできない

この問題は、対象アプリケーションでアプリケーション要素がどのように識別されるかに関する誤解によるものです。スパイツールは、ターゲット要素に関する情報（属性のリスト）を収集し、その情報をリストに表示します。

デフォルトでは、より有用な属性のいくつかが必須であるとマークされ、その他はマークされません。この選択は、要素を識別する際に適しているとわかっている属性の決定的なセットを表しておらず、単なる推測に過ぎません。リストを確認して、問題の要素に最適な属性セットを選択するのはユーザーが行う必要があります。

詳細については、「必須属性を選択する」を参照してください。

最後のテストを行ったときビジネスオブジェクトは動作していたが、オブジェクトを再ロードした後に動作しなくなった

対象アプリケーションをリロードするたびに、その要素の機能は劇的にも微妙にも変わることがあります。たとえば、ボタンのテキストが「更新」から「確定」に変更されたり（劇的な変更）、作成された要素の順序（序数として知られる）が変更された（微妙な変更）ことがあります。

ビジネスオブジェクト設計者として、アプリケーションが実行されるたびに同じままである属性のみを、必要に応じてマークするように注意する必要があります。序数が毎回異なる場合は、必須とマークしないことがおそらく最善です。

要素が常に変化するアプリケーションに直面した場合、採用できるさまざまな工夫があります。

  • 属性が予想どおりに変わった場合

    たとえば、ボタン上のテキストは、一部の他の変数に従ってある値から別の値に変わることがあります。この場合、テキストを動的にして要素を更新できます。

  • 属性が予想外に変化したものの、制限範囲内であった場合

    たとえば、要素の序数は変わることがありますが、常に2～4の範囲です。この場合、要素をコピーして貼り付けて複製し、各コピーの序数を新しい値に変更できます。ビジネスオブジェクトでは、CheckExistsクエリを含む待機ステージを使用して、それぞれの場合にフローを異なるポイントに分岐させることができます。

    序数を動的にすると、同様の効果が得られることに注意してください。

詳細については、「必須属性を選択する」を参照してください。

要素が設計時には正常に識別されるが、実行時には識別されない

この問題は、アプリケーションの実行中に一部の属性が変わることが原因と考えられます。

たとえば、要素がテキストを送る［テキスト］フィールドである場合、要素に書き込んだ後にそのテキストが変わるため、必須の属性としてテキストを使うべきではありません。

この問題により、［テキスト］フィールドに書き込むまでの最初の数分間はビジネスオブジェクトが機能し、その後は機能しなくなるという現象になります。

要素が無効または有効になったときに検出できない

この問題は、Microsoftアクティブアクセシビリティで使用されている用語で混乱が生じていることが原因である可能性があります。有効属性ではなく使用不可属性を使用してください。

デバッグ中、オブジェクトスタジオは引き続きフォーカスを合わせ続けています。対象アプリケーションにフォーカスを合わせる必要がある場合、これは問題になります。

以下の方法でオブジェクトスタジオの動作をコントロールできます。

  • タスクトレイのデバッグボタンを使用します（画面右下にあるPrismアイコンを探し、コンテキストメニューでオプションを選択します）。このメソッドはオブジェクトスタジオのフォーカスには影響しません。
ビジネスオブジェクトは1台のリソースPCで正常に機能しますが、別のPCでは機能しません

この問題では多くの原因が考えられます。基本的に、通常は対象アプリケーションが2番目のリソースPCで異なるように見えるか、動作が異なるためです。

こういった違いは、各対象マシンで使用されているソフトウェアのバージョンにまで及ぶ可能性があります（同じバージョンがすべてのリソースPCにインストールされていることを確認する必要があります）。または使用中のテーマ（例：Windows XPのテーマとWindowsクラシックのテーマ）は、アプリケーション要素のクラス名に非常に頻繁に影響することがあります。またはアプリケーションのレイアウトに影響するスクリーン解像度の違いなどハードウェアの違いもあります。

この問題を解決するには、すべてのリソースPCで対象の環境をできる限り統一するか、同じ属性を使用してそれぞれの異なる環境で識別できるように、要素属性を慎重に選択する必要があります。

後者の解決方法は広範囲なテストが必要となり、開発プロセスはより煩雑で難しいものになります。

デバッグ中、オブジェクトスタジオは引き続きフォーカスを合わせ続けています。対象アプリケーションにフォーカスを合わせる必要がある場合、これは問題になります。

以下の方法でオブジェクトスタジオの動作をコントロールできます。

  • タスクトレイのデバッグボタンを使用します（画面右下にあるPrismアイコンを探し、コンテキストメニューでオプションを選択します）。このメソッドはオブジェクトスタジオのフォーカスには影響しません。
アプリケーションが1つのウィンドウですべてのドキュメントを開くため、自分の新しいインスタンスを起動したり、スパイしたりできない

一部のアプリケーションは（たとえば新しい文書を開く際に）起動すると、必要な情報を収集し、この情報を既存のインスタンスに渡して終了します。こういったアプリケーションの1つにMicrosoft Wordがあります。

この問題に対する解決方法は、代わりにアプリケーションの既存のインスタンスにアタッチすることです。アタッチは事前または事後などいつでも行うことができますが、アプリケーションがすでに起動している必要があります。 アプリケーションをすでに実行しているかどうかを検出し、状況に応じて、起動または接続するロジックを実装する必要があります。

Internet Explorerのファイルダウンロードダイアログを操作できない

この問題は、Internet Explorerがローカルに構成されていることによるものです。 Blue Prismにお問い合わせいただき、Internet Explorerの設定について質問するか、次の回避策を使用してください。［アクティベート］アクションを使用してウィンドウを最前面に表示し、5秒間待ってからダウンロードダイアログの目的のボタンをクリックします。

Javascriptが機能しない（アクセスが拒否された）

Internet Explorerのバージョンや設定によっては、WebページでJavascriptを起動したときにさまざまな結果が生じることがあります。特にセキュリティ設定によっては、Javascriptメソッドを呼び出そうとすると 「アクセスが拒否されました」 というエラーが表示される場合があります。プロキシサーバーの設定は、Internet Explorerによるセキュリティ制限に影響を与える可能性があります。可能であれば、［設定を自動的に検出］チェックボックスをオフにしてください。これで、より厳格なルールが適用されるようになります。

アクティブアクセシビリティのパフォーマンスが遅い

Webページまたは複雑なアプリケーションで固有のアクティブアクセシビリティの要素を探そうとすると、パフォーマンスが低下することがあります。これは、一意の一致を探すためにテストする必要のある要素が非常に多数であることによります。MatchIndex属性に値を設定すると、パフォーマンスを上げることができます。これで、指定した数の一致が検出されると、Blue Prismは検索を中止するように指定できます。パフォーマンスは向上しますが、通常この属性を設定しない場合は、Blue Prismがオブジェクトモデル全体を検索することがわかります。Blue Prismが検索を実行し、一致が一意であることを確認することが重要です。一致が一意であることが確実な場合は、この属性を使用できます。

侵入型の手法が使用されている場合、Blue Prismは、接続しているアプリケーションで要素を見つけることができません。

アプリケーションモデルで「侵入型の手法」が有効になっている場合、アプリケーションにアタッチすると、Blue Prismのモデルで一部のコントロールにアクセスできない、または表示されないことがあります。

これは、侵入型の手法が有効になっていると、Blue Prismが添付アプリケーションで作成されているコントロールをリッスンするために発生します。Blue Prismがアタッチされる前にアプリケーションがコントロールを作成した場合、作成されたコントロールがあることを知る方法はなく、参照できません。

このコントロールを使用するには、アプリケーションをコントロールの作成前にアタッチするか、Blue Prismで問題のアプリケーションを起動し、すべてのコントロールに完全にアクセスできるようにする必要があります。
