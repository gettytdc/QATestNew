多组环境

用例

以下示例说明了如何在不同情况下使用流程、对象和资源组的访问权限。

专用流程访问

限制对流程的访问可以使其仅限特定用户可用

有三个流程团队：亚太地区开发人员、欧洲、中东和非洲开发人员和美国开发人员，每个团队都具有相同的安全角色权限集。每个团队负责其所在地区的流程。此外，还有一个在每个地区都承担一定责任的全球开发人员团队。

在 Studio 中，每个地区都有一个组，其中包含该地区开发人员所需的所有流程。目前，每个组的访问权限都不受限制，如图标覆盖所示。这意味着，访问权限由安全角色权限决定，并且尚未在流程组级别设置访问权限。

用户只需要访问与其地区相关的组中的流程。可通过更新每个组的访问权限来应用权限，以使用户无法访问其他地区的流程。将组设置为受限制，并删除不需要访问权限的用户角色的所有权限。以下示例显示了亚太地区流程组的访问权限，其中删除了美国开发人员角色的权限。

运行引用受限制对象的流程

用户尝试运行引用他们没有权限的对象的流程。

亚太地区开发人员角色对包含用于实现最终用户应用程序自动化的订单系统对象的文件夹没有访问权限。

亚太地区组中的流程引用订单系统对象。当具有亚太地区开发人员角色的用户打开流程时，将显示一个消息框，通知他们该流程包含对不可用项目的引用。这不会阻止它们打开和运行流程，但当流程尝试启动禁止的应用程序时会失败。

多个团队共享单个业务对象

允许不同地区的用户执行相同的业务对象

亚太地区开发人员、欧洲、中东和非洲开发人员和美国开发人员团队都需要在他们维护的流程中使用相同的业务对象。全球开发人员团队需要相关访问权限才能更新和维护对象。所有地区的开发人员角色必须至少为其角色启用“执行业务对象”和“执行流程”权限。全球开发人员角色需要 Process Studio 和 Object Studio 的完全权限。

在包含该对象的文件夹上设置访问权限，从而授予这三个地区的团队“执行业务对象”权限，并授予全球开发人员团队完全访问权限。

当亚太地区开发人员、欧洲、中东和非洲开发人员和美国开发人员团队运行其流程时，每个团队引用的共享业务对象可以成功运行。但是他们无法查看或编辑对象；只有分配了全球开发人员角色的用户才有权配置对象。

多个用户角色

如果由于用户分配了多个用户角色而发生权限冲突，则以最大的权限为准

某个用户是两个团队的成员：

  • 团队 1，具有流程 A 的查看流程定义权限
  • 团队 2，具有流程 A 的编辑权限

该用户具有完全访问权限，可打开和编辑流程 A，因为这是最大的权限。仅在团队 1 中的同事可以打开并查看流程 A，但无法对其进行编辑。

多个组

如果由于流程存在于多个组中而发生权限冲突，则以最小的权限为准

订单系统对象在受限制的默认组中，随后将其添加到不受限制的全局对象组中。订单系统对象继承了不受限制的全局对象的权限，因为其权限最小。

虽然默认组受到限制，但对象会显示不受限制的图标覆盖，因为权限继承自不受限制的全局对象组。
