トラブルシューティング データゲートウェイ

データゲートウェイシステムが正常に機能していることを確認する方法を教えてください。

データゲートウェイで問題を確認するには、次のようないくつかの方法があります。

  • コントロールルームでデータゲートウェイプロセスのステータスを確認する

    プロセスのステータスがエラーである場合、ステータスメッセージから、何が問題なのかについての情報が得られます。

  • アプリケーションサーバーでイベントログを確認する

    データゲートウェイプロセスのすべてのエラーは、Logstashプロセスマネージャーソースからイベントログに出力されます。Logstashプロセスの標準的出力ストリームは、エラーの場合、イベントログにも書き込まれ、Logstashの具体的なエラーを識別するのに役立ちます。

  • アプリケーションサーバーでLogstashログを確認する

    このログは「C:\Logstash\logstash\logs\」に書き込まれます。

データゲートウェイ専用のアプリケーションサーバーを使用する必要がありますか？

ほとんどのインストールでは、既存のアプリケーションサーバーをデータゲートウェイに使用できます。ただし、データゲートウェイを使用して大量のデータを処理したり、データを多数のターゲットに転送したりする場合は、この目的にのみ使用される専用のBlue Prismアプリケーションサーバーにデータゲートウェイをデプロイするのが適切な場合があります。

データゲートウェイプロセスがSQL Serverへの接続に使用するポートを変更できますか？

デフォルトでは、データゲートウェイのプロセスではポート1433を使用します。ただし、カスタム構成を編集して代替ポートを指定できます。

次の例に示すように、ポートは構成ファイル構造の［入力］セクションで定義されます。次のように、カスタム構成を編集し、強調表示されている値を必要なポート番号に置き換えます。

input {

jdbc {

jdbc_driver_library => "..\sqljdbc_4.2\enu\jre8\sqljdbc42.jar"

jdbc_connection_string => "jdbc:sqlserver://SQL_SERVER_INSTANCE:1433;databaseName=BP_

DATABASE;"

jdbc_driver_class => "com.microsoft.sqlserver.jdbc.SQLServerDriver"

jdbc_user => "<%Data Gateways BP Database SQL User.username%>"

jdbc_password => "<%Data Gateways BP Database SQL User.password%>"

statement => "delete top(3000)from BPADataPipelineInput with (rowlock, readpast)

output deleted.eventdata"

schedule => "*/3* * * * *"

}}
